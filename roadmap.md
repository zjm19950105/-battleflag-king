# 路线图

最后更新：2026-05-16

## 已完成

### Phase 1.0：Godot/C# 战斗原型

- Godot 4 + C# 项目可运行。
- JSON 数据加载、角色、装备、主动/被动技能基础模型完成。
- 1v1 / 3v3 自动战斗可运行。

### Phase 1.2：战前配置与 3v3

- 队伍组建、布阵、装备配置、被动配置、策略配置 UI 完成原型。
- 默认策略与敌方阵型数据化。

### Phase 1.3：被动技能与战斗阶段

- EventBus 接入战斗阶段。
- PendingAction 支持反击/追击/先制。
- DamageCalculation 允许被动介入。
- CC、Trait、Buff 生命周期、单步战斗 UI 完成原型。

### Phase 1.4-A：核心规则可信化

- A01-A16 已完成。
- 当前测试 135/135 通过。
- BattleEnd 被动结果语义已落地：BattleEnd 后治疗、HP 改变和 pending action 会参与最终胜负重算。
- `pas_rampage`、`pas_give_ap`、`pas_battle_horn` 已迁移到 structured `effects`，不再依赖 legacy tags。
- 结构化日志、金样例测试、数据契约、合规扫描已建立。

## 后续阶段

### Phase 1.5：6v6 竞技场

前提：Phase 1.4-B 的 UI 冒烟、关键拆分和金样例快照完成。

- 扩大战斗阵型到 6v6。
- 检查目标选择、Row/Column/All、性能和 UI 可读性。
- 不批量加角色，先用金样例职业组验证。

### Phase 2：大巴扎/爬塔循环

- 节点选择、奖励、招募、装备获得。
- 玩家策略/队伍持久化。
- 局外成长与失败结算。

####
这个游戏每一局对战都要消耗大量的精力去调整策略，所以应该在打赢对手之后有及时的高额的奖励。另外为了避免 玩家在一开始就故意认输，然后挑选想获得奖励的队伍。第一次打赢的奖励和后续打赢的奖励要有落差。
吃鸡的获得阵容中的角色，这是最高奖励。（阵容角色都有的话，获得随机角色）
反复调整才赢的阵容，33%概率获得角色，33%获得原石——3个原石可以合成一个角色，33%的几率获得某个角色的皮肤
UI上的属性界面还要调，六维图+偏肉盾等简介？就是角色的定位必须要靠美术+属性直观反应出来，原版游戏好像是加入了S这个属性评级，这样哪个属性突出一目了然——而且角色名字不能脸谱化，而是有自己独特的名字
####

## 长期架构方向

项目应保持“数据驱动 + 面向对象领域模型 + 函数式规则管线 + 事件驱动被动仲裁”。不要转向面向切面；不要让 Godot UI 成为规则宿主；不要为了未来假想扩展提前加接口。
