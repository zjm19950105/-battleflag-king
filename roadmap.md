## Phase 1：战斗核心验证（MVP）
**大目标**：实现圣兽之王竞技场的 6v6 编程自动战斗，可选人、可编程、可对战。

### 1.0 环境就绪（并行开展）
**目标**：能编译、能跑、能提交。
**任务**：
- Godot 4 + C# 项目创建
- 场景/节点/脚本基本操作熟练
- Git workflow（commit、branch、回滚）
- 基本的输入、渲染、打印调试跑通
**完结标准**：
- [x] 能 F5 跑起一个空白场景
- [x] 能在场景里画一个2D sprite并移动它
- [x] 能正常 commit 到 git

### >>> 1.1 1v1 普攻对战
**目标**：两个角色在极简战场上自动普攻到死。
**核心验证**：自动寻敌 → 移动 → 普攻 → 死亡 的最小闭环。
**任务**：
- 极简战场场景（两个出生点，足够距离）
- 角色数据：HP、攻击力、攻击速度、移动速度
- 自动行为：找最近的敌人 → 走到攻击范围 → 普攻 → 重复
- 死亡判定和战斗结束
**完结标准**：
- [x] 代码层闭环：BattleEngine + DamageCalculator + StrategyEvaluator 串联完毕
- [x] Main.cs 能加载数据并启动一场 1v1 战斗
- [x] 战斗结束有胜负判定（全灭 / AP耗尽按HP比例）
- [x] 控制台打印每回合战斗日志
- [x] CC框架接入：CharacterData/BattleUnit/Main.cs 支持CC状态切换，默认未CC
- [x] 12/12角色数据已填充（Subagent批量填充+修正），JSON格式验证通过
- [x] `CcInitialEquipmentIds` 配置：CC后新增装备槽强制带装备
- [x] `EquipmentSlot` 武器槽校验：MainHand永不为空
- [ ] 运行 Godot 场景（未CC状态），验证实际输出符合预期
- [ ] 一方 HP 归零，战斗结束

**【想法池】**
- 先把 1v1 代码闭环在 Godot 控制台跑通，再做 2D 视觉表现
- 当前使用 Lv50 数据导致 1v1 不平衡，后续可考虑缩放为 Lv1 等效值

### 1.2 1v1 技能对战
**目标**：引入第一个非普攻技能，验证数据驱动技能系统。
**核心验证**：技能配置（JSON）→ 读取 → 释放 → 效果。
**任务**：
- 技能数据结构（JSON Schema）：伤害、消耗、范围、效果类型
- 角色装备一个主动技能替代普攻（或技能+普攻共存）
- 技能释放逻辑：AP足够 → 敌人在范围 → 释放
- 一个简单技能效果（如：造成2倍伤害）
- 接入 EventBus：战斗开始/行动前/行动后触发被动技能
**完结标准**：
- [ ] 技能数据走 JSON，不改代码就能换技能参数
- [ ] 角色能按逻辑释放技能
- [ ] 被动技能在正确时机触发
- [ ] 技能效果（伤害/Buff/Debuff）正确生效

### 1.3 3v3 编程对战
**目标**：引入圣兽之王的核心——条件编程 + 多单位站位战术。
**核心验证**：玩家战前编程 → 战中自动执行。
**任务**：
- 战场扩展为6个位置（前排3、后排3），但只有3v3
- 角色AI系统：条件（If）→ 执行（Then）
- 首批条件：HP百分比、敌人距离、敌人类型
- 首批执行：攻击、后撤、待机、使用技能
- 优先级排序（多条指令按优先级执行第一条满足条件的）
- 射程概念：近战只能打前排，远程可以打后排
**完结标准**：
- [ ] 战前界面能给每个角色设置2-3条指令
- [ ] 战斗中角色按指令自动行动
- [ ] 前排/后排的射程、保护逻辑生效
- [ ] 战斗结果可复现（同样编程同样结果）

**【想法池】**
-

### 1.4 6v6 完整竞技场
**目标**：完整的选人 + 编程 + 对战体验。
**任务**：
- 角色池（至少4-6个不同角色/职业）
- 选人界面：从角色池选6人上场
- 编程界面：给6人分别设置完整指令集
- 更丰富的条件和执行指令
- 战斗表现优化（动画、特效可以 placeholder，但逻辑完整）
- 战斗日志/回放，方便调试编程逻辑
**完结标准**：
- [ ] 可选6人上场
- [ ] 完整的 if/条件/执行/优先级 编程系统
- [ ] 多种技能、职业有明确区分
- [ ] 战斗可运行、可观察、可调试
- [ ] **Phase 1 完结标志**：陌生人看战斗录像能说"这像圣兽之王的竞技场"

## Phase 2：大巴扎循环
**大目标**：把战斗嵌入 Roguelike 爬塔结构，每次前进=抽牌=成长。

### 2.1 最小地图循环
**目标**：战斗 → 奖励 → 再战斗。
**任务**：
- 极简节点地图（3-5个节点串联）
- 节点类型：战斗、休息（回血）、奖励
- 战斗奖励：获得一个新角色/技能/道具
- 角色/技能/道具的获取和装备逻辑
**完结标准**：
- [ ] 能走完整的一小轮地图
- [ ] 战斗胜利后能获得并装备新东西
- [ ] 新东西能影响下一场战斗

### 2.2 抽牌与天数系统
**目标**：完整的大巴扎式抽牌驱动流程。
**任务**：
- "抽牌"机制：每次前进从牌池抽一个节点
- 天数限制：总天数/行动点，耗尽则本轮结束
- 牌池设计：不同稀有度、不同效果
- 经济系统：金币/资源用于购买、升级
**完结标准**：
- [ ] 每次前进有抽牌感
- [ ] 天数压力形成决策张力
- [ ] 一轮游戏的开始→中期→后期有体验变化

### 2.3 局外成长（Meta）
**目标**：反复游玩有价值。
**任务**：
- 解锁新角色、新技能加入未来牌池
- 难度选择
- 成就/统计
**完结标准**：
- [ ] 玩完一轮后，下一轮有新内容

## Phase 3：本地化与合规
**大目标**：避免知识产权侵权，形成独立产品。

### 3.1 玩法剥离
- 圣兽之王的"编程自动战斗"是玩法机制，不受版权保护
- 但角色名称、世界观、美术、具体技能名称需要原创

### 3.2 资产原创
- 角色设计（名字、背景、外观）全部原创
- 技能名称、效果描述原创
- 地图/节点名称原创

### 3.3 代码与数据清理
- 确保没有引用原游戏的专有名词
- 数据表全部自查
