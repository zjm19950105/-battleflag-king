# 技能策略

最后更新：2026-05-11  
用途：记录《圣兽之王》原版“技能策略/作战条件”的文本结构与行为语义，作为本项目后续策略条件、目标选择和 UI 文案仿照依据。

## 设计目标

原版策略条件不是单纯的布尔开关，而是由“前缀 + 条件内容”组成。前缀决定条件是“优先排序”还是“硬性限定”；条件 1 与条件 2 同时设置时，整体遵循同时满足的组合语义。

本项目实现时应保持：

- JSON 存 canonical value，不存中文显示文本。
- 中文只在 UI/文案层出现。
- “优先”和“仅”必须在逻辑层区分，不要都实现成过滤条件。
- 新增策略条件时，同步更新 `docs/csharp-architecture.md`、`ConditionMeta`、`ConditionEvaluator` / `TargetSelector` 和对应测试。

## 原版文本结构

中文版原文的条件写法可以拆成两部分：

```text
前缀 + 内容
```

例：

```text
优先前排
仅步兵
优先HP最低
仅HP50%以下
```

## 两个核心前缀

| 前缀 | 原文例子 | 语义 | 行为 |
| --- | --- | --- | --- |
| 优先 | 优先前排 / 优先重甲 | 优先寻找这种目标 | 技能仍可发动；优先找符合条件的目标，找不到则按默认规则选择目标 |
| 仅 | 仅步兵 / 仅斥候 | 只有这种目标才打 | 条件满足才发动；没有合法目标时跳过本条技能 |

### 实现含义

- `优先` 更接近目标排序或目标偏好。
- `仅` 是硬过滤，也是技能是否可发动的一部分。
- 不应把 `优先HP最低` 误写成“只有 HP 最低时才发动”。
- 不应把 `仅骑兵` 误写成“优先骑兵，找不到打别人”。

## 条件 1 + 条件 2 逻辑

官方说明语义：

```text
可为每个技能最多设置 2 个条件，当满足所有条件时，角色将发动技能。
```

条件 1 和条件 2 是 AND 关系，不是 OR。

| 设置方式 | 行为 |
| --- | --- |
| 只设条件 1 | 满足条件 1 就触发 |
| 只设条件 2 | 满足条件 2 就触发 |
| 条件 1 + 条件 2 都设 | 两个条件都满足才触发 |
| 都不设 | 无条件触发，按技能默认目标规则选择目标 |

注意：这里的“满足所有条件”要结合前缀理解。`仅` 决定硬性可用范围，`优先` 决定目标选择顺序。

## 组合行为表

| 条件 1 前缀 | 条件 2 前缀 | 行为 |
| --- | --- | --- |
| 优先 | 空 | 优先找条件 1 目标；找不到则按默认目标规则 |
| 空 | 仅 | 条件 2 满足才触发；不满足则跳过本条技能 |
| 仅 | 仅 | 两个条件必须同时满足。例如 `仅斥候 + 仅步兵`，只有同时属于斥候与步兵的敌人才可作为目标 |
| 优先 | 仅 | 两个都生效。先找同时满足“优先”和“仅”的目标，再找满足“仅”的目标 |
| 优先 | 优先 | 先找同时满足两个优先条件的目标，再找条件 2 目标，最后找条件 1 目标 |

## 优先级细节

原版有一个容易踩坑的规则：

```text
优先前排 / 优先后排 无论放在条件 1 还是条件 2，都最优先执行。
```

例如：

```text
优先重甲 + 优先后排
优先后排 + 优先重甲
```

两者目标选择结果应等价：

```text
先找后排的重甲
再找后排
最后找重甲
```

因此，如果本项目要精确仿照原版，`前排/后排` 这类站位优先条件不能只按条件槽位顺序简单排序，需要有特殊的目标选择权重。

### 兵种类

```text
仅步兵 / 优先步兵
仅重甲 / 优先重甲
仅骑兵 / 优先骑兵

仅飞行 / 优先飞行
仅术士 / 优先术士
仅弓兵 / 优先弓兵
```

### 站位类

```text
仅前排 / 优先前排
仅后排 / 优先后排
仅前后排一列
优先前后排一列
优先人数最多的一排
优先人数最少的一排
仅2体以上的一排
仅3体以上的一排
```

`前后排一列` 用于可贯穿或列攻击技能，语义是寻找同一列中前排 + 后排都能命中的目标组合。
注意：原版日文条件中的“人数最多/最少的列”“2体以上/3体以上的列”按英文版和实战语义应理解为前排/后排这一“排”，不是位置 1-4、2-5、3-6 的纵列。

### HP / 属性类

```text
仅HP50%以下
仅自身HP≤75%
优先HP最低
优先HP最高
优先物防最高
优先魔防最低
```

### 状态类

```text
仅燃烧状态
仅中毒状态
```

状态类通常是硬限定：只有目标处于对应状态时才发动或才选择该目标。

### 特殊类

```text
第N次行动时
```

这类不是目标筛选，而是行动次数限定。实现时应归入发动条件，不应交给普通目标排序处理。

## 近战后排陷阱

近战物理技能通常不能越过前排直接攻击后排。策略条件不能违反技能本身的可达性。

因此：

- 不要给普通近战物理技能配置 `仅后排` 或 `优先后排` 作为期望命中后排的手段。
- 即使设置了后排条件，目标选择也必须先遵守技能射程、攻击类型、前排阻挡等基础规则。
- 远程、魔法、飞行、列攻击、贯穿等可以越排或扩展目标的技能，再结合对应条件处理。

| 概念 | 作用 |
| --- | --- |
| 条件槽位 | 条件 1 / 条件 2，最多两个 |
| 条件前缀 | `优先` 或 `仅`，决定排序还是硬过滤 |
| 条件内容 | 兵种、站位、HP、属性、状态、行动次数等 canonical 条件 |

建议内部结构类似：

```text
StrategyCondition
  - Mode: Preferred / Required
  - Category: Position / UnitClass / Hp / Status / AttributeRank / ActionCount / ...
  - Operator: equals / less_or_equal / highest / lowest / ...
  - Value: canonical value
```

目标选择顺序建议：

```text
1. 根据技能类型、目标阵营、攻击类型和前排阻挡，得到基础合法目标池。
2. 应用所有 Required 条件；如果没有目标，跳过该技能。
3. 对 Preferred 条件做排序。
4. 对“前排/后排”等原版特殊优先条件应用额外权重。
5. 按技能 target type 扩展 Row / Column / FrontAndBack 等多目标形状。
6. 如果 Preferred 条件找不到目标，但 Required 条件仍有合法目标，则回退到默认目标规则。
```

## 测试要点

后续实现或改动策略条件时，应至少覆盖：

- 只设置条件 1。
- 只设置条件 2。
- 条件 1 + 条件 2 同时设置时为 AND。
- `优先` 找不到目标时回退默认目标。
- `仅` 找不到目标时跳过技能。
- `优先 + 仅` 先找交集，再回退到满足 `仅` 的目标。
- `优先 + 优先` 先找交集，再按条件 2、条件 1 顺序或原版站位权重回退。
- `仅斥候 + 仅步兵` 这类多兵种交集。
- `优先前排 / 优先后排` 在条件 1 或条件 2 位置结果一致。
- 近战物理技能不能因为后排条件越过前排。
